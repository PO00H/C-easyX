
## 一、项目概述

**1. 项目名称** 盲剑客：《白夜瞬闪》

**2. 游戏类型** 像素动作 / 轻潜行 / 一击必杀

**3. 目标平台** 个人电脑

**4. 开发工具** C++ EASYX

**5. 项目周期** 二十天

**6. 项目目标**

- 在二十天内完成一款基于完全面向对象架构的高难度动作游戏核心版本；
    
- 重点实现基于波纹探测与记忆迷雾的非对称视野博弈、可破坏灯具系统，以及极致手感的蓄力一闪机制；
    
- 全流程包括：像素静态过场动画、三个递进难度的关卡地图、战斗结算与结局。

## 二、游戏核心概念与流程

**1. 核心世界观与静态过场** 主角曾是名震天下的剑术大师，在遭到家族残酷背叛与围剿时被夺去双眼。侥幸生还的他，在黑暗中觉醒了如同雷达般敏锐的听觉。 游戏启动后，通过三张像素风格的静态图片配合文字渐显，完成叙事：

- **第一幕：** 暴雨中的家族本庭，主角倒在血泊中，双眼被划伤，周围站满持刀的暗影；
    
- **第二幕：** 绝对纯黑的画面，伴随沉重的心跳声，屏幕中心突然亮起一圈微弱的白色波纹，代表听觉的觉醒；
    
- **第三幕：** 主角缠绕染血绷带的脸部特写，拔刀出鞘，屏幕闪过一道强烈白光，伴随利刃出鞘的音效，切入主菜单。
    

**2. 核心玩法机制**

- **波纹探测与记忆迷雾：** 玩家屏幕常驻黑暗，需主动释放声波探路。声波以玩家为中心向外扩散为一个圆形波纹，能够照亮一定半径内的地形与敌人。已被探测过的区域不会立刻变黑，而是以较低亮度“残影”形式在地图上留存，约 5 秒后逐渐淡回全黑。这种简化机制既保留了“盲人借声辨位”的叙事，又避免了复杂的射线检测与多边形绘制。
    
- **蓄力一闪系统：** 玩家的攻击具有极高的操作上限。鼠标仅提供方向指引，玩家按下攻击键后无法移动，进入原地蓄力前摇状态。蓄力时间越久，释放后向鼠标方向瞬间突进的距离越远。为降低严苛操作下的学习成本，在蓄力过程中，玩家脚下会依据当前蓄力值动态显示一个仅自己可见的落点预览圈；同时，在鼠标方向的突进终点位置显示一个白色「叉叉 UI」，直观地标记玩家蓄力达到当前等级后将会停靠的确切位置。若该终点落在**已被波纹探测过的墙壁**上，叉叉会变为红色并闪烁预警；若终点位于尚未探测的黑暗区域，则叉叉保持默认白色（未知即风险）。突进撞墙后玩家会被迫停下并进入**眩晕状态 1 秒**，期间无法移动与攻击，以此惩罚距离的误判，增加操作紧张感。

- **非对称视野博弈：** 主角是盲人，必须依赖主动探测；敌人视力正常，拥有扇形视野。若玩家在波纹照亮范围内暴露身影，或蓄力/释放声波时发出巨大声响，将引来敌人围攻。

- **一击必杀与打击反馈：** 敌我双方均无生命值，触之即死。斩杀判定成功时，触发零点一秒的全局游戏逻辑暂停，配合屏幕震动与向斩击反方向喷洒的血迹粒子，营造极致的斩断手感。

- **可破坏灯具与区域黑暗：** 特定房间内悬挂有灯笼/火把等照明灯具。灯具存在时，其影响范围内的敌人处于正常视野状态。玩家的突进若撞击灯具，则触发灯具破坏（玻璃/纸破碎音效 + 粒子），该灯具所属区域内的敌人立即进入深度黑暗状态。此时受影响敌人的视野半径归零，转为极度依赖听觉的“倾听/恐慌”状态——原地缓慢转身、小范围徘徊，并对任何声响高度敏感。灯具的明灭只影响敌人的视野状态，不改变玩家的视觉体验，玩家始终处于盲人视角，仅通过声波探测环境。该机制为每一间有灯的房间增加了策略选择：先灭灯还是先暗杀。

**3. 交互按键逻辑**

为降低操作学习成本并保证跨场景（菜单 ↔ 游戏）的一致性，项目采用以下统一输入方案：

- **WASD / 方向键：**
  - **游戏内：** 控制主角上/下/左/右移动。
  - **菜单/过场中：** 控制选项上下左右移动选择。

- **鼠标左键：**
  - **游戏内：** 长按进入蓄力状态，蓄力时间决定突进距离；松开左键立即释放突进。
  - **菜单/过场中：** 点击 UI 按钮确认选择。

- **回车键（Enter）：**
  - **菜单/过场中：** 确认当前高亮选项或推进对话/过场。
  - **游戏内：** 无作用（避免误触）。

- **空格键（Space）：**
  - **游戏内：** 主动释放一次波纹探测，以玩家为中心扩散圆形感知波。
  - **菜单中：** 无作用（或作为快捷返回）。

- **ESC 键：**
  - **任意场景：** 打开/关闭暂停菜单；在菜单最上层时返回上一级，直到返回主菜单。

这种设计将核心交互高度集中在 **WASD + 鼠标左键 + 空格** 三根手指上，菜单操作与主流 PC 游戏习惯完全一致，避免了复杂的组合键记忆。
    

**4. 三大核心关卡设计**

- **第一层：地底死牢。** 线性狭窄走廊，主要教学关卡。玩家在此熟悉声波探测、地形遮挡，并练习背后蓄力刺杀落单的守卫。本层不配置灯具，保持阴暗压抑的原始氛围。
    
- **第二层：外围竹林。** 开阔的非线性地图。存在大量可破坏的竹林障碍物，敌人巡逻路线交错，考验玩家在空旷地带的连续蓄力转移与多目标击杀。本层在哨站与路口选择性配置 2~3 盏灯具，引入环境策略深度。
    
- **第三层：修罗本殿。** 决战高台。掩体极少，包含弓箭手与视野极广的重甲守卫。玩家需要利用极致的听声辨位和极其精准的蓄力距离把控，清理全场。本层不配置灯具（或仅配置一盏装饰性灯具且不影响敌人视野），确保高难度不被逃课。

## 三、功能与底层架构需求分析

**1. 面向对象架构设计** 项目将彻底抛弃面条式代码，以类继承为骨架，同时预留组合（Component）接口，避免过深的继承链导致后期扩展困难。

- **实体基类：** 定义二维坐标、碰撞矩形、生存状态等通用属性。

- **动态实体类：** 继承自实体基类，增加移动速度、当前朝向向量。

    - **玩家控制类：** 封装键盘与鼠标的输入监听。内部维护一个状态机，负责处理空闲、移动、原地蓄力、突进、撞墙眩晕这五个状态的切换，以及声波发射逻辑。撞墙产生的 1 秒眩晕状态增加了操作失败的惩罚，让每次突进都伴随着真实的紧张感。

    - **敌军人工智能基类：** 封装视野扇形计算、巡逻路径节点数组，以及基于全局光照状态的视野切换。当灯具全灭进入深度黑暗时，视野归零并切换到 `PanicListen` 状态（原地缓慢转身、小范围徘徊、对声音极度敏感）。

        - **近战步卒类：** 发现玩家后，头上亮起红色惊叹号，执行直线追踪逻辑。

        - **远程弓手类：** 发现玩家后，试图拉开距离，并绘制一条预判射击的红色警戒线。

    - **环境道具类：**

        - **灯具类（`Lamp`）：** 带有矩形碰撞体的可破坏环境实体。被玩家突进撞击后播放破坏特效，并将所属区域的光照状态切换为 `DARK`。灯具破坏后本局不可恢复。灯具的明灭只影响敌人的视野状态，不影响玩家的视觉体验——玩家永远是“盲人视角”，灯具只是可被撞击的环境物体。

- **核心全局管理器（职责边界严格限定）：**

    - **关卡管理器：** 仅负责读取外部文本文件生成墙壁与地图障碍数据，不处理任何绘制或碰撞响应逻辑。

    - **特效管理器：** 仅统一维护顿帧计时器、屏幕震动偏移量、血迹粒子数组等“后处理与打击感状态”，不直接绘制任何实体。游戏主循环读取其状态后，再分别应用到渲染层。

    - **音频管理器：** 统筹加载音频文件，提供声音随距离衰减的播放接口。
        

**2. 核心系统逻辑交互**

- **波纹探测与记忆迷雾：** 玩家主动按下探测键时，以玩家坐标为圆心触发一个向外扩散的圆形波纹。波纹半径内的地形和敌人被立即“点亮”。同时，该圆形区域被加入“记忆迷雾”列表，以较低亮度绘制。每个记忆区域自带倒计时（约 5 秒），时间到后从列表移除，画面重新变黑。

- **碰撞与击杀判定：** 玩家在突进状态下，自身转化为一条带有宽度的线段，与敌人的圆形碰撞体进行线段与圆的相交检测，相交即判定斩杀。

- **突进撞墙判定：** 突进过程中若碰撞盒与墙壁相交，玩家立即停止于墙前，并进入 **1 秒眩晕状态**（无法移动、无法攻击）。若突进路径穿越灯具，灯具被破坏且玩家**正常继续滑行**，不会因此眩晕或中断。

- **灯具破坏判定：** 灯具为「可破坏环境体」，不参与「一击必杀」判定。玩家突进撞击灯具时，灯具被破坏，玩家突进将穿透灯具并按剩余距离继续滑行，直至最大突进距离或碰到墙壁/敌人。灯具不会杀死玩家，也不会被玩家突进以外的逻辑触发破坏。

- **听觉警报判定：** 玩家在蓄力阶段或释放声波时，会向四周广播一个听觉事件体，处于该半径内的敌人即使背对玩家，也会立即转身并进入警报状态。

**3. 资源管理与素材导入规范**

为避免资源路径混乱与内存泄漏，项目采用统一的 `assets/` 目录结构与集中式资源管理器。

- **目录结构：**
  ```
  assets/
  ├── images/
  │   ├── player/      # 玩家动画：待机、移动、蓄力、突进、斩杀
  │   ├── enemies/     # 敌人动画：近战步卒、远程弓手
  │   ├── effects/     # 特效：粒子、血迹、惊叹号、斩击线
  │   ├── backgrounds/ # 静态过场图与关卡背景大图
  │   └── tiles/       # Tiled 图块集（Tileset）
  ├── audio/
  │   ├── sfx/         # 短音效（斩击、脚步、受击、蓄力心跳）
  │   └── music/       # 背景音乐与章节主题曲
  ├── levels/          # Tiled 导出的地图文件（.tmx 或 .json）
  └── fonts/           # 像素字体文件
  ```

- **资源管理器：** 封装一个单例 `AssetManager`，在程序启动时统一调用 EasyX 的 `loadimage` 加载图片，音频使用 Windows 多媒体 API（如 `mciSendString` 或 `PlaySound`）统筹管理。游戏中所有实体只持有资源引用，禁止在业务逻辑中直接调用加载函数，防止重复加载与泄漏。

- **关卡地图格式：** 采用**纯文本字符矩阵**（`.txt`）或直接在代码中定义二维 `char` 数组描述关卡，每个字符代表一种地形或实体。例如：
  - `#`：墙壁碰撞体
  - `.`：可行走空地
  - `P`：玩家出生点
  - `E`：近战步卒
  - `A`：远程弓手
  - `B`：可破坏的竹林障碍物
  - `L`：灯具

  这种格式无需额外编辑器或 JSON 解析库，可直接在记事本中“画地图”，或在 C++ 代码中直接修改常量数组，是 EasyX 环境下最轻量的方案。

- **路径安全策略：** 代码中所有资源路径均基于运行时可执行文件（`.exe`）所在的目录进行拼接。开发时可将 `assets/` 文件夹放在工程输出目录下，确保在 IDE 调试与直接双击运行 exe 时都能正确加载素材。

- **开发期建议：** 美术迭代期间，只需替换 `assets/images/` 下的 PNG 文件，重新编译运行即可查看效果。建议先用纯色矩形或简单几何图形作为占位符，替核心机制跑通后再统一替换最终素材，避免前期在美术上耗费过多时间。

**4. 美术素材制作规范**

为适配 EasyX 的图像加载方式与项目整体像素风格，所有美术素材需遵循以下规范：

- **统一画布尺寸：**
  - 玩家与敌人单帧：32×32 像素，画布背景为透明（PNG 格式）。
  - 背景/过场图：按目标分辨率 960×540 或 1280×720 绘制，导出为 PNG。
  - 图块（Tileset）：单图块 32×32，按行或列排列成整张 Sprite Sheet。

- **动画帧图（Sprite Sheet）规范：**
  - 所有动画必须按顺序水平排列在同一张 PNG 中（一行内连续帧）。
  - 帧与帧之间不留空隙，总宽度 = 单帧宽度 × 帧数。
  - 文件命名：`[角色]_[动作]_[帧数]f.png`，例如：
    - `player_idle_6f.png`（玩家待机动画，6 帧）
    - `player_charge_attack_8f.png`（蓄力-突进动画，8 帧）
    - `enemy_melee_walk_4f.png`（近战步卒行走，4 帧）
    - `enemy_archer_alert_3f.png`（弓手警觉，3 帧）

- **各角色动画拆分清单：**
  - **玩家（盲剑客）：**
    - `idle`：待机呼吸（4~6 帧，循环）
    - `walk`：移动（4~6 帧，循环）
    - `charge`：蓄力姿态（2~3 帧，循环，可轻微抖动增加紧张感）
    - `dash`：突进残影（3~4 帧，单次播放，可用拖影表现高速）
    - `slash`：斩击定格（1~2 帧，单次播放，配合顿帧使用）
  - **近战步卒：**
    - `idle`、`walk`、`alert`（发现玩家后的警觉）、`chase`（追击）
  - **远程弓手：**
    - `idle`、`walk`、`alert`、`aim`（拉弓瞄准，2~3 帧）、`shoot`（射箭，2 帧）

- **环境道具素材：**
  - **灯具：** 一张 32×32（或 32×48）的灯笼/火把 PNG。可准备两帧轻微摇曳的循环（`lamp_idle_2f.png`），以及灯灭时的静态残骸图（`lamp_broken.png`）。特效部分（玻璃/纸破碎）可用程序粒子，无需额外动画帧。

- **特效素材：**
  - 血液粒子：一张 8×8 的红色/深红色圆点 PNG，程序中用代码控制数量与飞溅方向。
  - 惊叹号：一张 16×16 的红色 `!` 图标，敌人警觉时在头顶弹起并淡出。
  - 斩击线：一张细长白色渐变条（如 64×4），突进时沿方向拉伸并淡出来表现刀光。

- **文字与 UI：**
  - 使用像素字体（如 `Pixelify Sans`、`Zpix` 或 `Press Start 2P`），字号推荐 16px / 24px / 32px 三档。
  - 主菜单文字：标题使用 48px 加粗，选项使用 24px 常规。
  - 游戏内 HUD（如若有）：尽量简洁，蓄力进度用脚下预览圈代替，避免复杂 UI 破坏沉浸感。

- **背景与过场图：**
  - 三张过场图为静态像素插画，尺寸与游戏分辨率一致（推荐 960×540）。
  - 第一幕（暴雨本庭）色调：深蓝 + 暗红血迹，突出压抑感。
  - 第二幕（纯黑觉醒）色调：纯黑背景，中央白色声波圆环。
  - 第三幕（拔刀特写）色调：高对比度的绷带与刀光，最后接全屏白光切至菜单。
  - 关卡背景：因主画面常驻黑暗，背景图仅在被声波照亮时才可见，建议用低饱和度暗色绘制，细节不宜过多。

## 四、时间节点与进度规划

将项目分为四个阶段，每个阶段五天。

**第一阶段：底层基建与蓄力手感原型（第一天至第五天）**

- **第一天：** 搭建开发环境，建立基础工程目录，确立所有类的头文件结构。头文件设计阶段需为后期灯具机制预留接口：`class Lamp : public Entity;` 的空壳声明、`enum class LightState { BRIGHT, DARK };` 的定义，以及 `EnemyAI` 基类中 `virtual void OnLightStateChanged(LightState state);` 的虚函数接口。
    
- **第二天：** 实现玩家控制类的基础移动与鼠标指向角度计算。
    
- **第三天：** 开发核心的蓄力一闪机制，实现按下按键停止移动、计算蓄力值、松开后按鼠标方向瞬移的基础逻辑。
    
- **第四天：** 编写特效管理器，接入顿帧与屏幕震动逻辑，用基础方块测试打击手感。
    
- **第五天：** 整合音频模块，加入蓄力音效与斩杀音效，完成第一个战斗原型里程碑。
    

**第二阶段：波纹探测系统与第一关卡（第六天至第十天）**

> 本阶段核心从复杂的射线检测降级为更易实现的圆形波纹探测 + 记忆迷雾。这是为了保证项目在 20 天内能够顺利落地的关键决策。

- **第六天：** 实现玩家主动释放的圆形波纹探测逻辑。波纹以玩家为中心向外扩散，一定半径内的墙壁和敌人被临时“点亮”。
    
- **第七天：** 实现记忆迷雾系统。被波纹照亮过的区域以较低亮度留存，并设置约 5 秒的倒计时，时间结束后从记忆中移除，画面回归黑暗。
    
- **第八天：** 编写关卡管理器，读取字符矩阵（代码中常量数组或 `.txt` 文本）生成墙壁碰撞体与实体出生点。
    
- **第九天：** 搭建第一层地底死牢地图，测试玩家在狭窄地形中的波纹探测与记忆迷雾表现。
    
- **第十天：** 优化波纹扩散的动画表现，完成视觉渲染里程碑。
    

**第三阶段：非对称人工智能与关卡铺设（第十一天至第十五天）**

- **第十一天：** 编写敌军基类，实现扇形视野计算与被声波照亮时的显形逻辑。
    
- **第十二天：** 实现近战步卒类的状态机，完成巡逻与听声追击逻辑。在该日同时实现“可破坏灯具”核心逻辑与敌人 `PanicListen` 状态。
    
- **第十三天：** 搭建第二层外围竹林地图，引入可被玩家斩断的竹子障碍物类，并在地图中铺设首批灯具测试策略效果。
    
- **第十四天：** 开发远程弓手类，并搭建第三层修罗本殿地图。
    
- **第十五天：** 打通三层地图的连贯切换机制，要求清理完当前层所有敌人才能进入下一层。
    

**第四阶段：叙事包装与打磨发布（第十三至二十天穿插进行）**

- **第十三天起（穿插进行）：** 制作开场三段式像素过场动画及主菜单界面。建议尽早打通“主菜单 → 进游戏 → 返回菜单”的完整循环，以提升开发时的心理完成度与整体流畅感。
    
- **第十六天：** 优化所有音效的时机，增加斩杀时的血液粒子喷射系统。
    
- **第十七天：** 性能优化，排查内存泄漏问题，优化波纹探测与记忆迷雾列表在多敌人场景下的绘制开销。
    
- **第十八天：** 全流程封闭测试，反复调整玩家蓄力距离与敌人视野的数值平衡，确保难度平滑。
    
- **第十九至二十天：** 最终代码清理，编写代码注释，打包生成可执行文件，完成项目。

## 五、项目执行与管理要点

**1. 逻辑与渲染严格分离** 在面向对象设计中，必须保证所有数据计算先行，画面绘制居后。顿帧效果通过停止逻辑更新的时间步长来实现，但不能停止画面的刷新。

**2. 核心手感的优先级** 第一阶段的蓄力一闪手感是整个项目的灵魂。如果顿帧时间和震动幅度调试不到位，宁可推迟后续地图的开发，也必须将核心战斗体验打磨到极致。

**3. 音频的叙事作用** 对于一款以盲人为主角的游戏，音频不仅是反馈，更是玩法。必须确保声音能准确反映玩家动作的危险程度，蓄力的前摇声音应该是一种极具压迫感的低频音效，以增强玩家操作时的心理紧张感。

**4. 技术风险与架构预留** 波纹探测系统是本项目的核心技术，但相比射线检测已大幅简化，关键是确保“记忆迷雾”的列表管理不泄漏、不卡顿。同时，AI 行为与地图铺设工作量大、迭代次数多，建议为 Enemy 基类预留简单接口，后期可快速扩展。若开发过程中发现时间吃紧，可优先减少动画帧数、使用纯色占位图、甚至将第三层地图缩小，优先保证核心手感与通关流程完整。
